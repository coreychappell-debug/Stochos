## Stochos NY Pipeline Rebuild and Automation Procedure

### Purpose

This procedure rebuilds and automates the New York pipeline on the Stochos server.

At the end of this procedure, the server will:

1. Refresh New York raw data into `/srv/stochos/data/raw/new_york/nylottery_data`
2. Normalize that data into DuckDB at `/srv/stochos/data/duckdb/stochos_lottery.duckdb`
3. Run both steps automatically every week through cron

The current normalized pipeline uses:

raw sales staging table
raw retailer staging table
dimension tables for games and retailers
a normalized fact table
a unified modeled view for revenue, estimated payouts, commissions, and contribution logic 

---

## Part 1. Understand the two automated functions

### Function A: Raw NY data refresh

This job downloads or refreshes the New York source files and stores them under:

`/srv/stochos/data/raw/new_york/nylottery_data`

This is the ingestion layer.

### Function B: NY DuckDB normalization refresh

This job reads the raw files and builds the normalized warehouse model inside DuckDB.

The current Python normalization script:

reads the raw sales CSV and raw retailer CSV
builds staging tables
builds retailer and game dimensions
melts the wide sales table into a long fact table
builds a unified modeled view and reconciliation outputs 

This is the warehouse layer.

## Part 3. Permissions setup

This is important. If permissions are wrong, the jobs may work manually for one user but fail later under automation.

### Step 3.1 Create or confirm the shared group

```bash
groupadd stochos
```

If the group already exists, Linux will say so. That is fine.

### Step 3.2 Add analyst users to the group

Replace usernames if needed, but based on your current environment these are the relevant people:

```bash
usermod -a -G stochos analyst1
usermod -a -G stochos analyst2
usermod -a -G stochos coreychappell
```

### Step 3.3 Set group ownership and inheritance

```bash
chgrp -R stochos /srv/stochos
chmod -R g+rwX /srv/stochos
find /srv/stochos -type d -exec chmod g+s {} \;
```

This does three things:

1. makes the `stochos` group own the directory tree
2. gives the group read/write access
3. makes new files inherit the `stochos` group automatically

### Step 3.4 Ensure the DuckDB folder is writable

```bash
chmod 2775 /srv/stochos/data/duckdb
ls -lah /srv/stochos/data
```

You want to see the `duckdb` folder with permissions like:

`drwxrwsr-x`

### Step 3.5 If the RStudio container writes as uid 1000

In your environment, the RStudio process wrote as uid 1000. You verified this with:

`whoami = rstudio`
`uid=1000(rstudio)` in the container session

If ACL support is needed again in a rebuild, install ACL tools:

```bash
apt-get update
apt-get install -y acl
```

Then allow uid 1000 to write into the DuckDB folder:

```bash
setfacl -m u:1000:rwx /srv/stochos/data/duckdb
setfacl -d -m u:1000:rwx /srv/stochos/data/duckdb
getfacl /srv/stochos/data/duckdb
```

You should see an ACL line for `user:1000:rwx`.

This was necessary because the RStudio runtime user inside the container was not a normal host user, but still needed write access.

## Part 10. Set up weekly cron automation

This is the actual automation layer.

### Step 10.1 Open root’s crontab

```bash
crontab -e
```

If you are asked for an editor, choose `nano` unless you prefer another editor.

### Step 10.2 Add these lines

Use this schedule as the baseline:

```cron
0 2 * * 1 /usr/bin/python3 /srv/stochos/jobs/ny_refresh.py >> /srv/stochos/logs/ny_refresh.log 2>&1
30 2 * * 1 /usr/bin/python3 /srv/stochos/jobs/ny_duckdb_refresh.py >> /srv/stochos/logs/ny_duckdb_refresh.log 2>&1
```

What these mean:

First line
Every Monday at 2:00 AM, refresh raw NY data

Second line
Every Monday at 2:30 AM, run the DuckDB normalization refresh

## Part 12. Important operational cautions

### DuckDB lock conflicts

DuckDB allows one writer at a time.

If the Python refresh job fails with a lock error, it means something else has the database open for writing. In your environment this happened because an RStudio `rsession` was holding the DuckDB file open.

If needed, close the connection in R:

```r
dbDisconnect(con, shutdown = TRUE)
rm(con)
gc()
```

Best practice:

Python is the canonical writer
R is a read-only consumer whenever possible

### RStudio memory issue

You originally discovered that large NY transforms could crash RStudio due to out-of-memory conditions. The kernel explicitly killed `rsession` during large in-memory reshaping. That is why the ETL logic was moved to Python plus DuckDB. The warehouse script exists to avoid repeating that failure mode.

### Date parsing

During the build process, one major issue came from date format assumptions in `BUS_DAY`. The corrected pipeline now explicitly handles parsing so that fact rows are not silently filtered to zero. This was one of the most important debugging fixes in the rebuild history.

### Quotes and SQL string safety

Another major issue was unescaped apostrophes in SQL string generation, such as game names containing single quotes. The script now includes helper quoting functions to avoid SQL parser failures.
